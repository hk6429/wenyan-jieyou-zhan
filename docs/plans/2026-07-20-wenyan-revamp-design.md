# 文言解憂站改版設計規格 — 全套八角理論強化計畫

## 背景與目標

孩子回饋文言解憂站沒有姊妹站「字鬥英雄」(vocab-duel) 好玩。診斷結果：本站剛完成的內容擴充（27 篇全注譯、27 人對戰、23 張立繪、逐段三分頁）把「學習內容」做厚了，但**長線黏著機制仍停在第一代**——只有 PvE 刷血條＋文豪錄收藏，缺少字鬥英雄「詞靈二期／三期」引入的**即時對戰、融合、市場、城市養成**四項社交黏著與空間感機制。

本次改版目標：比照字字珠璣同日排定的四大子系統計畫，把四項機制**改寫成文言文主題的版本**，並用 Yu-kai Chou 八角理論 (Octalysis 8 Core Drives) 逐項檢查，把主要驅動力拉到 9 分——不是換皮照搬，而是利用文言文題材獨有的優勢（名句、對聯、科舉、文人知音傳統）做出差異化。

## 隱喻對應總表（字字珠璣 → 文言解憂站）

| 字字珠璣原案 | 文言解憂站版本 | 對應理由 |
|---|---|---|
| 即時對戰 (rtbattle) | **文戰擂台**（房號約戰＋戰帖＋全班文會＋科舉賽季） | 機制直接移植；賽季稱號改用科舉功名（童生→秀才→舉人→貢士→進士→狀元），比字鬥英雄的抽象段位更有題材記憶點 |
| 字靈書院 (shuyuan) | **解憂草堂**（庭園養成＋卷軸藏書閣＋名句題聯） | 685 顆字珠改為「27 卷軸＋學習量裝飾」；創造力玩法改為**從已讀選文的名句選句組聯**——文言文站做對聯是天然優勢，字鬥英雄完全做不到 |
| 神獸融合 (fusion) | **文魄合契**（兩位精通文豪結契喚醒文魄） | 「融合出新神獸」改為「兩位文豪心志相通、喚醒共同精神的文魄」——貼合文人「知音」傳統，情感敘事比神獸血脈更貼題 |
| 翰墨集市 (market) | **文房市集**（筆墨紙硯道具班級交易） | 交易物改為文房四寶道具（裝飾草堂＋小幅對戰加成）；防作弊、限流、班級限定機制全數沿用 |

## 使用者確認事項（沿用字字珠璣的三項共識）

1. 全套一次排（文戰擂台＋文魄合契＋文房市集＋解憂草堂），非單項優先
2. 接受文言解憂站從純靜態站升級為輕量後端（Cloudflare Pages Functions + D1）
3. 新美術資產跟改版同步排產（codex exec 生圖，Q版國風水墨、米黃宣紙＋朱紅印章，對齊既有 27 張立繪）

## 技術架構總覽

**統一後端**：新增 Cloudflare Pages Functions + D1，`_kv.js` D1 shim（Redis 風格指令層）直接複製字字珠璣／字鬥英雄已驗證的寫法。文戰擂台、文魄合契、文房市集共用同一組後端，但 **HMAC 金鑰與資料表命名空間分離**（`wy_rt_*` / `wy_fusion_*` / `wy_market_*`），避免單點外洩波及多系統。

**⚠️ 開工第一天就要做對的地雷**（vocab-duel 踩過）：後端只集中在 Cloudflare Pages 一個平台，前端一律透過統一的 `WYAPI` helper 打**絕對網址**，讓 vercel.app／netlify.app 鏡像站的後端功能不降級。禁止相對路徑 `fetch('api/room')`。

**與現有引擎的介面**：`js/battle.js`（純函式核心：`newBattle`／`resolveAnswer`／combo 封頂）與 `js/quiz.js` 出題器**完全不用改**。即時對戰讓雙方各自在本機用 `resolveAnswer` 打「我方 vs 標靶」，把輸出傷害同步給對方當血量扣減來源——「同 seed 不同機、伺服器只當狀態郵筒」模式，題庫與戰鬥運算全部留在前端。

**新增底層（兩個子系統的前置依賴）**：
- **墨錠貨幣**：文言解憂站目前沒有任何貨幣。新增 `WYStore` 的 `inkIngots` 欄位，答對掉落（含 combo 加成小幅提高掉率），供文魄合契（合契費）與文房市集（交易幣）共用。規則頁明寫「墨錠不可兌換現實金錢或禮物」。
- **班級碼**：文言解憂站目前無班級概念。新增選填 `classCode`（比照字字珠璣 `selfstudy.classCode`），文房市集與全班文會依賴它做班級限定。

## 四大子系統設計

### 1. 文戰擂台（即時對戰）

- **配對機制**：4 位數房號＋10 分鐘 TTL（同步對戰）；另加 **6 碼戰帖＋7 天 TTL**（非同步版：下戰帖的一方先打完留下逐題成績，應戰方任何時間用同一 seed 打同一組題，逐題比對放榜）——成本極低、社交黏著不輸同步版，一併做
- **同步方式**：1.5 秒輪詢（非 WebSocket），伺服器只存進度與心跳，不存題目；出題靠雙方本機用同一亂數種子跑 `WYQuiz`
- **選題範圍**：開房者指定（單篇／某難度層 J·S／混合 27 篇），seed＋scope 一起下發，保證兩機題目與選項順序完全相同
- **文言解憂站獨有差異化——「硯靈事件」**：字字珠璣接的是奇遇系統，本站沒有奇遇層，改為由 seed 決定每 5 題觸發一次「硯靈評點」事件（雙倍傷害、下一題排除一個錯誤選項、combo 門檻變動），效果只影響觸發方；事件台詞用文言腔（「硯靈曰：此子可教，賜墨勢倍之」），實作為獨立小模組 `rt-events.js`，不動 battle.js
- **全班文會**：老師開房、全班同題搶答＋即時排名，**只露前五**＋自己名次（白帽設計，避免公開處刑）；戰報由「硯靈宣讀」包裝
- **科舉賽季**：月為週期重置排位，積分換算功名稱號（童生／秀才／舉人／貢士／進士／狀元），賽季結束發榜動畫＋可分享戰報卡；每日對戰次數上限（稀缺性，防沉迷）
- **任務骨架**（對應字字珠璣 rtbattle 計畫的 11 個 Task，可直接沿用其後端程式碼）：
  1. `WYAPI` 共用 API helper（絕對網址、跨子系統契約）
  2. D1 kv/hash/list/zset schema＋假 D1 測試 helper
  3. `_kv.js` D1 shim
  4. `rt-room` 後端（create／join／push／poll）
  5. rtbattle 純邏輯層（同 seed 出題、傷害記帳、勝負判定、斷線判勝）
  6. 前端房間流程＋對戰同步 UI
  7. 硯靈事件整合（seed 決定事件序列）
  8. 非同步戰帖（6 碼＋7 天 TTL）
  9. 全班文會（老師開房全班同題）
  10. 科舉賽季排位（週期重置＋功名稱號）
  11. 部署與端到端驗證（雙分頁真機互打）

### 2. 解憂草堂（庭園養成視覺層）

- **佈局**：山水庭園自由式佈局，中軸「文氣十境」山門對應全站總精通度十階；左右延展**三院落依 J／S 難度與題型進度 derive 繁茂度**（蒙學院＝國中 14 篇、觀止軒＝高中 13 篇、藏書閣＝總答題量）
- **收藏實體化**（字字珠璣是 685 顆字珠，本站的血肉是選文與學習量）：
  - 每篇達成精通（答對率 ≥80% 且 ≥8 題）→ 藏書閣多一幅**掛軸**（該篇題名＋文豪小像）
  - 累積答對題數換算庭園裝飾：竹叢、蓮池、松石、錦鯉，數量門檻分品階
  - 對戰勝場換算：石燈籠、演武樁
- **創造力差異化（全計畫最重要的超車點）——名句題聯**：匾額與對聯**不開放自由輸入**（風險控管沿用原案），但詞庫不是預設清單，而是**從玩家已精通選文的名句池選句組聯**——精通〈陋室銘〉才能把「斯是陋室，惟吾德馨」掛上門楹；讀得越多、可用的句子越多。這把「賦予創造力」和「真實學習量」直接鎖死掛鉤，符合養成鐵律，也是字鬥英雄與字字珠璣都做不到的題材優勢
- **自由擺放**：預設散佈＋拖曳座標存檔＋還原（沿用字字珠璣 Task 4 的實作）
- **慶典與成就牆**：解鎖新境界／新院落時慶典動畫（epicShow 手法）；成就牆陳列連對紀錄、精通篇數、科舉功名
- **任務骨架**（對應原案 9 個 Task）：store 基座（`wy_caotang`）→ 山門十境＋三院落 derive → 學習量裝飾實體化 → 自由擺放 → 名句題聯（詞庫＝精通篇名句池）→ 慶典佇列＋成就牆 → UI 骨架 → 互動 → 慶典動畫
- **純前端、不依賴新後端**——投報率最高，可最先動工

### 3. 文魄合契（融合系統）

- **敘事**：兩位文豪若心志相通，可在草堂設壇「合契」，喚醒兩人共同精神所化的**文魄**——貼合「知音」傳統（伯牙子期），比神獸血脈更貼文言文題材
- **觸發門檻**：兩位文豪所屬選文皆達精通（答對率 ≥80%）且各累積 ≥30 題作答量；合契消耗墨錠
- **雙親不消耗**（沿用字鬥英雄 P3-11 血淚教訓）：合契後兩位文豪原樣保留，文魄是新增收藏，不是替代
- **失敗機制走溫和路線**：兩成機率失敗，只扣墨錠＋安慰台詞（文言腔：「文氣未合，且再讀書」）＋回饋少量墨錠；文豪與學習進度完全不受影響；墨錠每日獲取有上限＝失敗的代價是時間而非資產（白帽原則）
- **文魄全庫封頂 6 隻**（控制生圖產量），配對舉例：
  - 隱逸之魄：陶淵明（飲酒／桃花源記）×袁宏道（晚遊六橋待月記）
  - 忠義之魄：諸葛亮（出師表）×岳飛（滿江紅）
  - 豪俠之魄：木蘭（木蘭詩）×虯髯客（虯髯客傳）
  - 史筆之魄：司馬遷（鴻門宴）×李斯（諫逐客書）
  - 曠達之魄：蘇軾（記承天寺夜遊×赤壁賦，同人雙篇特例）
  - 鄉土之魄：鄭用錫（勸和論）×洪繻（鹿港乘桴記）
- **配方揭曉解謎（未知性）**：合契前不公開文魄長相；答對一題**跨兩篇的隱藏綜合題**才解鎖配方揭曉
- **被動二選一（創造力）**：文魄喚醒時給兩個被動效果讓玩家挑（如「隱逸：對戰失誤一次免反擊」vs「淡泊：墨錠掉落 +10%」），不是系統配好
- **文魄隨行出戰**：選一隻文魄隨行，效果走 battle 外掛層（新增薄薄一層 `fusion-adapter`，不動 battle.js 核心）
- **暱稱＋文魄名片**：自訂暱稱＋canvas 渲染分享卡
- **任務骨架**（對應原案 9 個 Task）：墨錠底座＋掉落鉤子 → 合契資格判定 → 合契核心（成功路徑）→ 失敗機制 → 被動二選一 → 配方揭曉解謎 → 文魄隨行出戰 → 暱稱＋名片 → 合契壇 UI＋smoke

### 4. 文房市集（市場交易系統）

- **交易物限定文房道具，文豪與文魄不可交易**——「文豪是師友不是商品」，保留情感資產
- **道具庫**：筆墨紙硯四類（湖筆／徽墨／宣紙／端硯等品階分層），來源＝對戰勝利與精通里程碑掉落；用途＝裝飾草堂＋小幅對戰加成（走 fusion-adapter 同一條外掛通道）
- **班級限定預設開啟**（依賴新增的 classCode）：同班可見可交易，全站公開另設開關；保留單走同班名單驗證
- **伺服器端防作弊全數沿用原案**：驗貨（品項白名單＋價格帶檢查）、HMAC 簽章、claimKey 領貨款、10% 交易稅（墨錠回收）、每日限購 3 件、限流
- **每週五限時開市**（稀缺性）＋珍品每週限量
- **社交白帽補強**：購買自動附「感謝小卡」（預設選項、不開放自由文字）；集市達人 `stars` 排行
- **風險控管（照抄原案五條，一條不減）**：
  1. 排除任何真錢加值，規則頁明寫墨錠不可兌現
  2. 不開放自由文字輸入，全走預設選項，杜絕留言霸凌管道
  3. 預設班級限定，暱稱驗證與限流沿用字鬥英雄機制
  4. 定價公開透明（依品階設價格帶），不做隱藏演算法折扣
  5. 共用 D1 命名空間與金鑰分離
- **任務骨架**（對應原案 9 個 Task）：後端純邏輯核心（驗貨・價格帶・時窗・簽章）→ post／list＋限流 → buy（限購・簽章・感謝小卡）→ cancel／claim（10% 稅）→ 保留單＋珍品限量＋stars → 前端邏輯層 → 瀏覽 UI → 上架購買流程 → 排行＋收藏冊＋部署

## 八角理論總表（文言解憂站版）

| 驅動力 | 現況初評 | 主要負責子系統 | 拉到 9 分的關鍵補強 |
|---|---|---|---|
| **1. 意義使命** | 3 | 合契（文魄）、草堂 | 文魄＝喚醒文豪共同精神的知音敘事；草堂境界升階觸發文豪書信劇情段落 |
| **2. 成就感** | 4 | 擂台（科舉）、草堂 | 科舉發榜動畫＋可分享戰報卡；成就牆陳列功名與精通掛軸 |
| **3. 賦予創造力** | 2 | **草堂（2→9，全站最弱也是最大機會）** | 名句題聯（詞庫＝已精通選文名句池）＋自由拖曳擺放＋被動二選一——文言文站的天然題材優勢，兩個姊妹站都做不到 |
| **4. 擁有感** | 5 | 合契（8）、市集（→9）、草堂（→9） | 文魄自訂暱稱＋名片；文房道具品階收藏；掛軸藏書閣＋個人庭園 |
| **5. 社交影響** | 1 | **擂台（1→9）**、市集 | 房號約戰＋戰帖＋全班文會（只露前五）＋分享卡；市集感謝小卡＋達人排行 |
| **6. 稀缺性** | 3 | 擂台、合契、市集 | 每日對戰上限＋科舉賽季；文魄全庫 6 隻＋墨錠日上限；週五開市＋珍品限量 |
| **7. 未知性** | 4 | 擂台（硯靈事件）、合契 | seed 硯靈事件（每 5 題隨機評點）；配方揭曉解謎（跨篇隱藏題） |
| **8. 損失規避** | 3 | 合契、市集 | 一律時間成本（墨錠日上限、失敗多等）取代資源扣減；市集維持溫和設計 |

**設計原則**：損失規避相關補強一律優先時間成本而非資源扣減——教育產品白帽原則，不為衝分數犧牲孩子的挫折耐受度。

## 建議實作優先順序

1. **解憂草堂基礎迴圈**（山門十境＋三院落＋掛軸藏書閣＋名句題聯）——不依賴新後端、資料底子已厚（27 篇精通度、立繪、答題統計都是現成的），投報率最高
2. **文戰擂台核心**（WYAPI＋D1 shim＋房號配對＋輪詢＋seed 出題）——技術路徑已被兩個姊妹站驗證，後端程式碼可大量複用
3. **硯靈事件＋全班文會＋戰帖**——差異化賣點與教室場景，優先於市集/合契
4. **文魄合契核心迴圈**（墨錠底座 → 合契 → 隨行出戰）——先跑通結構，解謎與名片分批補
5. **文房市集**——放最後：風險評估最複雜，且需要後端防作弊機制最成熟後再上線
6. **八角補強細節**（分享卡、賽季、慶典動畫、時效性代價）穿插在對應子系統完成後補上

## 美術資產排產

codex exec 生圖產線分批執行（沿用既有 SOP：雙線以上並行、單張 200s timeout、落盤驗證、避免單批撞用量上限），風格對齊既有 27 張立繪（Q版國風水墨、米黃宣紙、朱紅印章）：
